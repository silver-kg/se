## 1. 서론

### 1.1 문서의 목적

본 프로젝트 관리 계획서는 **Preedom (가짜 일정으로 행동을 앞당기고, 자유를 만든다)** 개발 프로젝트의 전반적인 관리 활동을 체계적으로 수행하는 것을 목적으로 한다.


- 프로젝트 개요
- 개발 계획 및 환경
- 팀 구성
- 품질 관리
- 산출물


### 1.2 프로젝트 개요

#### 1.2.1 프로젝트 정의
 
사용자의 미루기 행동을 줄이기 위해 실제 마감 이전에 가상의 마감 기한을 생성하여 작업 수행을 유도하는 것을 목적으로 한다.


실제 마감일보다 앞선 가짜 마감일을 자동 생성해 미루기를 방지하고, 작업 수행 데이터를 기록해 사용자가 자신의 패턴을 확인할 수 있도록 한다.

#### 1.2.2 프로젝트 목표
- 사용자의 미루기 습관을 효과적으로 개선하는 시스템 구현
- 단순한 일정 관리가 아닌, 행동 유도와 자기 인식을 동시에 지원
- 직관적이고 사용하기 쉬운 사용자 인터페이스 제공
- 정확한 행동 데이터 수집 및 분석을 통한 개인화된 인사이트 제공
- 2027년 4월까지 시스템 개발 완료 및 베타 테스트 진행
#### 1.2.3 기능적 요구사항

- **FR-001**: 사용자는 작업명, 실제 마감일, 예상 소요 시간을 입력하여 새로운 작업을 등록할 수 있어야 한다
- **FR-002**: 시스템은 실제 마감일보다 최소 1일 이상 앞선 가짜 마감일을 자동으로 계산하여 표시해야 한다
- **FR-003**: 사용자는 작업 시작/일시정지/재개/완료 버튼을 통해 실시간으로 작업 수행을 기록할 수 있어야 한다
- **FR-004**: 시스템은 시간대별(오전/오후/저녁) 집중도 통계를 그래프로 시각화하여 제공해야 한다
- **FR-005**: 사용자는 "계획한 시간 vs 실제 소요 시간" 비교 차트를 작업별로 확인할 수 있어야 한다
- **FR-006**: 시스템은 과거 유사 작업 데이터를 기반으로 새로운 작업의 예상 소요 시간을 추천해야 한다
- **FR-007**: 사용자는 주간/월간 단위로 작업 완료율, 평균 집중 시간, 가장 생산적인 시간대 등의 리포트를 조회할 수 있어야 한다
- **FR-008**: 시스템은 가짜 마감일 D-3, D-1, D-Day에 알림(푸시/이메일)을 발송해야 한다

## 2. 개발 계획

### 2.1 프로젝트 모델 및 개발 방법론

**선정된 프로세스 모델: Agile - Scrum 모델**

**선정 이유:**
- 요구사항이 명확하지 않고 사용자 피드백을 기반으로 기능을 점진적으로 개선해야 함
- 가짜 마감 생성 알고리즘과 행동 패턴 분석 로직은 실제 사용자 데이터를 통해 지속적으로 조정 필요
- 짧은 스프린트(2~4주)를 통해 프로토타입을 빠르게 제작하고 검증 가능
- 팀원 간 의사소통이 원활하게 이루어져 빠른 의사결정 가능
- 사용자의 요구사항 변경에 유연하게 대응 가능

**개발 방법론: 객체지향 방법론**

**채택 이유:**
- 작업, 사용자, 마감 일정, 분석 결과 등을 객체로 모델링하여 관리 용이
- 캡슐화를 통해 각 모듈의 독립성 확보 및 유지보수 효율성 증대
- 클래스 단위로 모듈화하여 단위 테스트 및 확장 용이
- 재사용성이 높아 향후 AI 기반 분석 기능 등 확장에 유리

## 3. 팀 구성

### 3.1 팀 구조
**하이브리드 팀 구조 (Scrum 기반)**: PM/PL이 전체 방향을 관리하고 개발팀은 자율성을 가지고 Sprint를 수행하는 수평적 구조

#### 개발팀 (6명)

- UI/UX 디자이너 (1명)
- 프론트엔드 개발자 (2명)
- 백엔드 개발자 (2명)
- 데이터 분석 개발자 (1명)


### 3.2 역할 및 책임

| 역할 | 담당자 | 주요 책임 |
|------|--------|-----------|
| **프로젝트 관리자 (PM)** | 김가은 | 프로젝트 전반 관리, Product Backlog 관리, 이해관계자 커뮤니케이션 |
| **프로젝트 리더 (PL)** | 김가은 | Scrum 프로세스 촉진, Sprint 주관, 일정 모니터링 |
| **형상/품질 관리자** | 김가은 | 버전 관리, 테스트 계획 수립 및 수행, 품질 검증 |
| **UI/UX 디자이너** | 김가은 | 사용자 경험 설계, 화면 설계 및 프로토타입 제작 |
| **프론트엔드 개발자** | 김가은 | React 기반 프론트엔드 개발, UI 컴포넌트 구현, 데이터 시각화 |
| **백엔드 개발자** | 김가은 | Node.js 기반 API 개발, 시스템 아키텍처 설계, 데이터베이스 연동 |
| **데이터 분석 개발자** | 김가은 | **가짜 마감 생성 알고리즘 개발**, 행동 패턴 분석, 데이터 수집/전처리 |

- 참석자: 팀장, 팀원

#### 4.1.4 Sprint Retrospective:
- Sprint 종료 시 1시간
- 프로세스 개선 논의
- 참석자: 팀장, 팀원


### 4.2 변경 사항 관리 방법
- 기능 변경 시 팀원 전체의 권한 및 다수결 합의 필요
- 변경 사항은 GitHub Issue에 기록
- Sprint Planning에서 변경 사항 논의 및 승인
- 승인된 변경 사항만 Product Backlog에 반영

### 4.3 산출물 관리 방법

**파일 명명(Naming) 방법**

| 카테고리 | 형식 | 예시 |
|----------|------|------|
| 소프트웨어 | preedom-v{버전}-{날짜}.zip | preedom-v1.0.0-20270430.zip |
| 문서 | [Preedom]_{문서명}_YYMMDD_Doc-{버전}.md | [Preedom]_프로젝트관리계획서_260426_Doc-001.md |
| 그 외 파일 | YYMMDD_{목적}_{버전}.{확장자} | 260426_분석알고리즘_v1.py |

**버전 관리 방법 및 담당자**
- **소스 코드**: GitHub를 통해 관리, Git Flow 브랜치 전략 사용 (담당자: 전체 개발팀)
- **문서**: 파일명에 날짜 기재, 문서 내 제/개정 이력 수정 (담당자: 해당 작성자)
- **저장소**: 모든 소스 코드는 **GitHub**에 업로드, 문서는 GitHub 또는 클라우드 공유
